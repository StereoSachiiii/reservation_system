## 🚀 Impressive Features

### 1. 🎯 Spatial Scoring Algorithm (V6)
The most sophisticated part of the backend is the **Visibility & Proximity Scoring Engine** located in `StallService.java`.
- **Dynamic Valuation**: Instead of static pricing, the system calculates a `Visibility Score` (0-100) for every stall.
- **Influence Zones**: Stalls gain value based on proximity to "Influences" (Entrances, Stages, Main Walkways).
- **Exponential Falloff**: The closer a stall is to the center of an influence, the higher its score, with support for `EXPONENTIAL` falloff curves.
- **Narrative Pricing**: The API returns a `Value Drivers` breakdown (e.g., "+15 Stage Proximity"), which tells vendors *why* a stall costs more.

### 2. 🎨 Interactive Stall Designer
Located in `StallDesignerPage.tsx`, this is a custom-built canvas editor for event organizers.
- **Percentage-Based Layout**: Geometry is stored as percentages (0-100) rather than pixels, ensuring the map looks identical on a 4K monitor or a mobile phone.
- **Dynamic Scoping**: The designer allows hall-specific overrides. You can drag and drop "zones" and "influences" which then feed directly into the pricing engine.
- **Stall Intelligence**: Stalls aren't just rectangles; they carry metadata for `Category` (Retail, Premier, etc.) and `Size` which are used for real-time validation.

### 3. 🎫 Automated Entry Pass System
A full-stack workflow for reservation lifecycle:
- **QR Generation**: High-performance QR generation using `ZXing` in `QrService.java`.
- **JWT-Secured Downloads**: Confirmation emails contain a direct link to download the pass, secured by a short-lived **JWT token in the query string**.
- **In-Line Embedding**: Emails are sent with the QR code embedded as a CID (Content-ID), so it shows up even without an internet connection in most mail clients.

## 🤨 Out of Pocket (Noteworthy & Unusual)

### 1. 🌌 The 10 Billion Threshold
Found in `StallDesignerPage.tsx:L123`:
```typescript
id: s.id > 10_000_000_000 ? undefined : s.id
```
The system distinguishes between "persisted" and "new" stalls by checking if their ID is greater than **10 Billion**. This is a creative (if slightly terrifying) way to handle client-side temporary IDs without collisions.

### 2. 🔍 JSON-String Grep for Coordinates
Found in `StallService.java:L178`:
```java
if (geomStr.contains("\"x\": 0") || geomStr.contains("\"x\": 0.0")) {
    calculatedScore -= EDGE_DISTANCE_PENALTY;
}
```
Instead of parsing the JSON to check the X coordinate, the backend uses a partial string match. If your stall starts at `0.0`, it gets a penalty for being a "Wall Hugger".

### 3. 🧩 The "Dummy" Coordinate Space
Found in `StallDesignerPage.tsx:L139`:
```typescript
width: 1000, height: 600, // Dummy pixel space representation
```
The entire spatial engine maps physical percentages to a hardcoded `1000x600` grid. It works perfectly, but it's a bold choice for a "dynamic" system.

### 4. 💰 Cent-Based Pricing Constants
Found in `application.properties`:
```properties
app.pricing.stall.small=500000
app.pricing.stall.medium=1000000
app.pricing.stall.large=2000000
```
The system avoids floating point errors by storing everything in cents. A Medium stall costs 1,000,000 "units", which is likely handled as 10,000.00 in the UI.

## 🛠 Project Architecture Summary

- **Backend**: Spring Boot 3 + JPA. Feature-rich service layer emphasizing logic over boilerplate.
- **Frontend**: React 18 + Vite. Heavy use of Context API (`DesignerContext`) for complex state management.
- **Database**: PostgreSQL. Availability is derived at query time, keeping the DB clean and reducing race conditions.
